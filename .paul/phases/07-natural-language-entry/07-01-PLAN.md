## Goal
Add a backend NL parsing pipeline: Convex stores ephemeral parse requests, the bridge processes them via OpenClaw with a focused parsing prompt, and results flow back through Convex for Swift client consumption via subscription.

## Purpose
Phase 7 requires Loom-powered NL parsing for event and task creation views (without opening the chat panel). This plan builds the backend infrastructure that Swift clients will call in Plan 07-02.

## Output
- `parse_requests` Convex table for request/response lifecycle
- `convex/nlParse.ts` with mutations and queries
- `convex/http.ts` extended with bridge-facing parse endpoints
- Bridge updated to poll and process parse requests via OpenClaw
</objective>

## AC-1: Parse Request Created in Convex
```gherkin
Given a Swift client sends a mutation with text "Dentist Thursday 3pm" and type "event"
When the nlParse:createRequest mutation executes
Then a parse_requests document is created with status "pending", the text, type, and a client-generated requestId
And the document has a createdAt timestamp
```

## AC-2: Bridge Detects and Processes Parse Requests
```gherkin
Given a parse_requests document with status "pending" exists in Convex
When the bridge polls /pending-parse
Then the bridge receives the pending request
And the bridge calls OpenClaw with a focused NL parsing system prompt (no chat history)
And the bridge posts the structured result to /parse-result
```

## AC-3: Parse Result Stored and Queryable
```gherkin
Given the bridge has posted a parse result for a requestId
When a Swift client subscribes to nlParse:getResult with that requestId
Then the query returns the parsed result with status "complete"
And the result contains structured fields (title, date, time, duration for events; title, priority, dueDate for tasks)
```

## AC-4: Error Handling and Timeout Resilience
```gherkin
Given OpenClaw is unreachable or returns an error
When the bridge attempts to process a parse request
Then the bridge sets the request status to "error" via /parse-result
And the Swift client receives the error status and can fall back to manual entry
```

    **1. Add `parse_requests` table to `convex/schema.ts`:**

    Add after the `studio_events` table definition:
    ```typescript
    parse_requests: defineTable({
      requestId: v.string(),                          // client-generated UUID
      text: v.string(),                               // raw NL input
      type: v.union(v.literal("event"), v.literal("task")),
      status: v.union(v.literal("pending"), v.literal("complete"), v.literal("error")),
      result: v.optional(v.string()),                 // JSON string of parsed fields
      createdAt: v.int64(),                           // UTC milliseconds
    })
      .index("by_request_id", ["requestId"])
      .index("by_status", ["status"]),
    ```

    **2. Create `convex/nlParse.ts`:**

    Exports:
    - `createRequest` (mutation, internal): accepts `{ requestId: string, text: string, type: "event" | "task" }`, inserts a document with status "pending" and createdAt = Date.now()
    - `updateResult` (mutation, internal): accepts `{ requestId: string, status: "complete" | "error", result?: string }`, patches the matching document
    - `getResult` (query, public): accepts `{ requestId: string }`, returns the document matching the requestId (or null). This is what Swift subscribes to.
    - `createParseRequest` (mutation, public): thin wrapper around createRequest that the Swift client calls directly. Accepts `{ requestId, text, type }`.
    - `cleanup` (mutation, internal): deletes parse_requests older than 5 minutes — called opportunistically, not by cron (to avoid cron bloat)

    Follow the patterns in `convex/chatMessages.ts` for internal vs public exports.

    **3. Extend `convex/http.ts` with two new endpoints:**

    `GET /pending-parse` — Returns the oldest pending parse request (status = "pending"). Bridge calls this on each poll cycle. Returns `{ pending: false }` if none, or `{ pending: true, requestId, text, type }` if one exists. Uses `ctx.runQuery(internal.nlParse.oldestPending, {})`. Add `oldestPending` as an internal query in nlParse.ts that returns the first pending request ordered by createdAt.

    `POST /parse-result` — Bridge posts `{ requestId, status, result }`. Calls `ctx.runMutation(internal.nlParse.updateResult, { requestId, status, result })`. Also opportunistically calls cleanup to purge old requests.

    Both endpoints use the same optional LOOM_WEBHOOK_SECRET auth pattern as existing endpoints.

    Avoid:
    - Do NOT add a cron job for cleanup — opportunistic cleanup in /parse-result is sufficient
    - Do NOT add real-time subscriptions to the bridge — polling is fine
    - Do NOT modify existing tables or endpoints

    **Add parse request handling to the bridge's poll loop.**

    **1. Add a `checkParseRequests()` function:**

    ```javascript
    async function checkParseRequests() {
      const headers = { "Content-Type": "application/json" };
      if (WEBHOOK_SECRET) headers["Authorization"] = `Bearer ${WEBHOOK_SECRET}`;

      const res = await fetch(`${CONVEX_SITE_URL}/pending-parse`, { headers });
      if (!res.ok) return;

      const { pending, requestId, text, type } = await res.json();
      if (!pending) return;

      console.log(`[bridge] Parse request: "${text}" (${type})`);

      // Build focused parsing prompt (no chat history, no calendar context)
      const systemPrompt = buildParsePrompt(type);

      const currentPassword = process.env.OPENCLAW_PASSWORD || process.env.OPENCLAW_TOKEN || OPENCLAW_PASSWORD;

      try {
        const loomRes = await fetch(`${OPENCLAW_URL}/v1/chat/completions`, {
          method: "POST",
          headers: {
            "Content-Type": "application/json",
            "Authorization": `Bearer ${currentPassword}`,
          },
          body: JSON.stringify({
            model: "openclaw",
            messages: [
              { role: "system", content: systemPrompt },
              { role: "user", content: text },
            ],
          }),
        });

        if (!loomRes.ok) {
          console.error(`[bridge] Parse OpenClaw error: ${loomRes.status}`);
          await postParseResult(requestId, "error", null);
          return;
        }

        const data = await loomRes.json();
        const replyText = data.choices?.[0]?.message?.content ?? "";

        // Extract JSON from reply
        const parsed = extractJSON(replyText);
        if (parsed) {
          await postParseResult(requestId, "complete", JSON.stringify(parsed));
        } else {
          console.error("[bridge] Could not extract JSON from parse response");
          await postParseResult(requestId, "error", null);
        }
      } catch (err) {
        console.error(`[bridge] Parse error: ${err.message}`);
        await postParseResult(requestId, "error", null);
      }
    }
    ```

    **2. Add `buildParsePrompt(type)` function:**

    For type "event":
    ```
    You are a natural language parser for calendar events. Parse the user's text into a structured event.

    Current date/time: {now in USER_TIMEZONE}
    User's timezone: {USER_TIMEZONE}

    Respond with ONLY a JSON object (no markdown, no explanation):
    {
      "title": "event title (cleaned of date/time words)",
      "start": "ISO 8601 datetime in user's timezone",
      "duration": 60,
      "isAllDay": false
    }

    Rules:
    - If no time specified, set isAllDay to true and start to the date at midnight
    - If no date specified, assume today (or tomorrow if current time is past the mentioned time)
    - Default duration is 60 minutes unless specified
    - Extract the title by removing date/time references from the input
    - Always use timezone {USER_TIMEZONE}
    ```

    For type "task":
    ```
    You are a natural language parser for tasks. Parse the user's text into a structured task.

    Current date/time: {now in USER_TIMEZONE}
    User's timezone: {USER_TIMEZONE}

    Respond with ONLY a JSON object (no markdown, no explanation):
    {
      "title": "task title (cleaned of date/time/priority words)",
      "priority": "medium",
      "dueDate": "ISO 8601 datetime or null if no due date mentioned",
      "hasDueTime": false
    }

    Rules:
    - Priority: "high" if words like urgent/important/critical/asap, "low" if words like maybe/sometime/low-priority, "medium" otherwise
    - If a date but no time is mentioned, set hasDueTime to false
    - If a specific time is mentioned, set hasDueTime to true
    - Extract the title by removing date/time/priority references
    - Always use timezone {USER_TIMEZONE}
    ```

    **3. Add `extractJSON(text)` helper:**

    Try to parse the entire text as JSON first. If that fails, find the first `{` and last `}` and try to parse that substring. Return the parsed object or null.

    **4. Add `postParseResult(requestId, status, result)` function:**

    POST to `${CONVEX_SITE_URL}/parse-result` with `{ requestId, status, result }`. Same auth pattern as other Convex posts.

    **5. Integrate into poll loop:**

    In the `poll()` function, add `await checkParseRequests()` at the top, BEFORE the pending messages check. This ensures parse requests are handled even when no chat messages are pending. Parse requests should not set `processing = true` (they're independent of chat flow).

    Avoid:
    - Do NOT modify the existing chat message handling logic
    - Do NOT add calendar/task context to parse prompts — keep them focused and fast
    - Do NOT change the poll interval — 2s is fine for parse latency
    - Do NOT add new environment variables — reuse OPENCLAW_URL, OPENCLAW_PASSWORD, USER_TIMEZONE
