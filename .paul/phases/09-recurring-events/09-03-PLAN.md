<objective>
## Goal
Extend NotificationService to schedule local notifications for recurring event occurrences within the 48-hour notification window, not just the master event's original start date.

## Purpose
Currently, recurring events only get a notification for the master's first occurrence. A daily standup created for Monday won't notify on Tuesday or Wednesday because NotificationService receives raw subscription events (masters only). Virtual occurrences are expanded in CalendarViewModel but never reach NotificationService.

## Output
- NotificationService.rescheduleEventNotifications expands recurring events and schedules notifications for each occurrence within the 48-hour window
</objective>

<task type="auto">
  <name>Task 1: Expand recurring events in NotificationService</name>
  <files>LoomCal/Services/NotificationService.swift</files>
  <action>
    In `rescheduleEventNotifications(_ events: [LoomEvent])`:

    1. Before the `for event in events` loop, build an expanded list:
       ```swift
       let expandedEvents = expandForNotifications(events)
       ```
       Then iterate over `expandedEvents` instead of `events`.

    2. Add a private helper method `expandForNotifications(_ events: [LoomEvent]) -> [(event: LoomEvent, fireStart: Date)]`:
       - For each event:
         - If NOT recurring (rrule is nil): add `(event, startDate)` where startDate is from event.start ms
         - If recurring:
           - Parse rrule with `RecurrenceRule.from(rrule:)`
           - Get exception dates from `event.exceptionDateValues`
           - Call `rule.occurrences(from: masterStart, through: cutoff, excluding: exceptions)`
             where cutoff = now + 48 hours
           - For each occurrence date, add `(event, occurrenceDate)` to the result
           - Use the master's duration/title/etc for all occurrences
       - Return the full list

    3. Update the scheduling loop to use the expanded list:
       - Replace `for event in events` with `for (event, startDate) in expandedEvents`
       - Remove the inline `startDate` computation (already provided)
       - Use `"\(prefix)\(event._id)_\(Int(startDate.timeIntervalSince1970 * 1000))"` as the notification identifier
         (includes timestamp to differentiate occurrences of the same master)
       - Keep the same content, trigger, and fireDate logic

    Avoid: Changing the method signature — callers already pass [LoomEvent].
    Avoid: Expanding beyond 48 hours — match the existing cutoff window.
    Avoid: Scheduling for all-day recurring events (existing guard continues to skip them).
  </action>
