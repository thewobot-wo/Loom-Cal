**convex/schema.ts** — Extend the `chat_messages` table:
- Add `v.literal("pending_action")` to the `role` union: `v.union(v.literal("user"), v.literal("assistant"), v.literal("pending_action"))`
- Add `action: v.optional(v.string())` — JSON string of the action payload
- Add `actionStatus: v.optional(v.union(v.literal("pending"), v.literal("confirmed"), v.literal("cancelled"), v.literal("undone")))` — tracks lifecycle of action cards

**convex/chatMessages.ts** — Add two new functions:

1. `writePendingAction` (internalMutation) — args: `{ content: v.string(), action: v.string() }`. Inserts a chat_messages row with `role: "pending_action"`, the display text as `content`, the action JSON as `action`, `actionStatus: "pending"`, and `sentAt: BigInt(Date.now())`.

2. `updateActionStatus` (mutation) — args: `{ id: v.id("chat_messages"), actionStatus: v.union(v.literal("confirmed"), v.literal("cancelled"), v.literal("undone")) }`. Calls `ctx.db.patch(id, { actionStatus })`. This is called from iOS when user taps Confirm/Cancel.

Return `_id`, `title`, `start`, `duration`, `timezone`, `isAllDay`, `location`, `calendarId`, `taskId`. Import `internalQuery` from "./_generated/server".

**convex/tasks.ts** — Add `listForLoom` (internalQuery, no args). Fetch all tasks where `completed === false`. Return `_id`, `title`, `dueDate`, `priority`, `hasDueTime`, `completed`, `notes`. Import `internalQuery` from "./_generated/server".

**convex/http.ts** — Add two new routes (keep existing /loom-reply and /pending-messages):

1. `GET /loom-context` — HTTP action that:
   - Checks optional LOOM_WEBHOOK_SECRET auth (same pattern as existing endpoints)
   - Runs `ctx.runQuery(internal.events.listForLoom, {})`, `ctx.runQuery(internal.tasks.listForLoom, {})` in parallel with `Promise.all`
   - Returns JSON: `{ events: [...], tasks: [...] }`
   - Import `internal` from "./_generated/api" (already imported)

2. `POST /loom-pending-action` — HTTP action that:
   - Checks optional LOOM_WEBHOOK_SECRET auth
   - Parses request body: `{ displayText: string, action: object }`
   - Calls `ctx.runMutation(internal.chatMessages.writePendingAction, { content: displayText, action: JSON.stringify(action) })`
   - Returns `{ ok: true }` on success, 400 on invalid body

**bridge/loom-bridge.mjs** — Major extension (keep existing structure, extend the `poll()` function):

1. Add `extractAction(replyText)` function:
   - Regex: `/ACTION:\s*```json\s*([\s\S]*?)```/`
   - If match found: parse JSON, strip ACTION block from display text, return `{ action, displayText }`
   - If no match or parse fails: return `{ action: null, displayText: replyText.trim() }`

2. Add `fetchContext()` function:
   - `GET ${CONVEX_SITE_URL}/loom-context` with auth headers
   - Returns parsed JSON `{ events, tasks }`
   - On failure, returns `{ events: [], tasks: [] }` (degrade gracefully)

3. Add `buildSystemPrompt(context)` function that formats the system prompt per the research doc's Pattern 9:
   - Tells Loom it is a calendar and task assistant
   - Lists current events (next 7 days) in readable format: "- [title] on [date] at [time] ([duration]min) [id: _id]"
   - Lists active tasks: "- [title] (priority: [priority], due: [date or 'no due date']) [id: _id]"
   - Defines the ACTION JSON format with all 6 action types: create_event, update_event, delete_event, create_task, update_task, delete_task
   - Includes the displaySummary field requirement
   - Specifies previousValues for edits
   - Specifies ambiguity rules: list numbered options if multiple match, default to all-day if no time, create with what's available
   - Reinforces that Loom proposes (shows preview), never executes directly
   - Default calendarId to "personal" for new events

4. Add `postPendingAction(displayText, action)` function:
   - `POST ${CONVEX_SITE_URL}/loom-pending-action` with JSON body `{ displayText, action }`
   - On failure, fall back to `postLoomReply(displayText)` (existing function, refactored from inline code)

5. Refactor `poll()`:
   - After detecting pending messages, call `fetchContext()` to get current calendar/task state
   - Build system prompt with `buildSystemPrompt(context)`
   - Forward to OpenClaw with system message prepended: `[{ role: "system", content: systemPrompt }, ...messages]`
   - After receiving reply: call `extractAction(replyText)`
   - If action found: call `postPendingAction(displayText, action)`
   - If no action: call existing reply post (refactor the inline fetch into a `postLoomReply(content)` function)

6. Extract the existing reply-posting code into a reusable `postLoomReply(content)` function.

IMPORTANT: Keep the bridge dependency-free (no npm packages, just native fetch). Keep the POLL_INTERVAL and other env vars unchanged. The system prompt should instruct Loom to include the Convex document `_id` when referencing existing events/tasks for edits/deletes. Timestamps in ACTION payloads should be string-encoded milliseconds (e.g., `"1708988400000"`).
