## Summary

Phase 3.1 is a retroactive gap-closure phase, not a feature phase. The v1.0 milestone audit found four documentation/integration gaps left from Phases 2 and 3. All gaps are small: one is a documentation file rename/creation, one is a checkbox traceability update, one is a two-line Swift code change in two view files, and one is adding a frontmatter field to a summary document.

The most substantive code change is wiring `isTimeBlock: event.taskId != nil` in both `WeekTimelineView.weekEventView()` and `DayTimelineView.eventCards()`. `TimelineEventCard` already accepts and fully implements the `isTimeBlock` parameter — it was built in Plan 03-04. TodayView already passes `isTimeBlock: item.event.taskId != nil` correctly. The gap is that the other two views that render events (`DayTimelineView` and `WeekTimelineView`) were not updated when the parameter was added in Plan 03-04.

The Phase 2 verification file exists at `.planning/phases/02-calendar-views/VERIFICATION.md` but must be renamed to `02-VERIFICATION.md` to match the success criterion ("02-VERIFICATION.md exists"). Alternatively, the success criterion may be satisfied by creating a new `02-VERIFICATION.md` that summarizes or references the existing file — the exact approach is at planner discretion.

**Primary recommendation:** Four discrete, non-overlapping tasks: (1) rename Phase 2 verification file, (2) wire `isTimeBlock` in WeekTimelineView and DayTimelineView, (3) add `requirements-completed` frontmatter to 03-02-SUMMARY.md, (4) confirm CALV-02 checkbox state in REQUIREMENTS.md.

### WeekTimelineView Gap (current, custom inline view — no TimelineEventCard used)

```swift
// Source: LoomCal/Views/Calendar/WeekTimelineView.swift, lines 234-263
// weekEventView() renders events as custom Button with hardcoded Color.blue styling
// It does NOT use TimelineEventCard at all — it has its own inline layout
private func weekEventView(event: LoomEvent, columnWidth: CGFloat) -> some View {
    let yPos = yOffsetForStart(event.start)
    let eventHeight = max(CGFloat(event.duration) / 60.0 * pointsPerHour, 16)

    Button(action: { onEventTap(event) }) {
        HStack(spacing: 0) {
            RoundedRectangle(cornerRadius: 1)
                .fill(Color.blue)               // ← hardcoded blue, never orange
                .frame(width: 2.5)
            // ...
        }
        .background(
            RoundedRectangle(cornerRadius: 3)
                .fill(Color.blue.opacity(0.12)) // ← hardcoded blue
        )
    }
```

**Key finding:** WeekTimelineView's `weekEventView()` does NOT use `TimelineEventCard`. It has its own custom inline event rendering with hardcoded `Color.blue`. There are two valid approaches:

**Option A: Switch to TimelineEventCard**
Replace `weekEventView()` with a `TimelineEventCard` instance. This achieves full parity and reuses the canonical card, but the narrow column layout in week view uses a constrained `columnWidth - 3` width and 10pt font (vs the card's full layout). The card already handles narrow columns gracefully (it clips text). The `isTimeBlock` parameter would be passed naturally.

**Option B: Inline color switch in weekEventView()**
Keep custom `weekEventView()` but add `let isTimeBlock = event.taskId != nil` and switch colors:
```swift
let accentColor: Color = event.taskId != nil ? Color.orange.opacity(0.8) : Color.blue
let bgColor: Color = event.taskId != nil ? Color.orange.opacity(0.12) : Color.blue.opacity(0.12)
```

**Recommendation: Option B (inline color switch)**. WeekTimelineView columns are narrow (as little as ~40pt at compact sizes) and use a compact 10pt font layout specifically tuned for width. `TimelineEventCard` uses a `subheadline` font and shows a task icon — potentially too large for narrow columns. The inline approach matches the existing narrow-column threshold logic in the view (line 244: `if columnWidth > 50 { Text(event.title) }`). Option B is lower risk and preserves the narrow-column optimization.

Locked decision says "Claude's discretion on whether to show a task icon alongside the event title" — given narrow columns, the icon is best omitted in week view. The orange accent bar alone is sufficient visual distinction.

## Common Pitfalls

### Pitfall 1: Renaming VERIFICATION.md Breaks References

**What goes wrong:** Other documents (ROADMAP.md, STATE.md, success criteria) may reference `VERIFICATION.md` by the old name.
**Why it happens:** File rename without grep for existing references.
**How to avoid:** After renaming, grep for `VERIFICATION.md` to find any broken references and update them.
**Warning signs:** Any docs that link to the old path.

### Pitfall 2: WeekTimelineView Uses Custom Inline Rendering (Not TimelineEventCard)

**What goes wrong:** Planner assumes WeekTimelineView uses TimelineEventCard and writes a task to "pass isTimeBlock to TimelineEventCard" — the parameter cannot be passed to a view that isn't used.
**Why it happens:** The Phase 3 verification docs say "isTimeBlock param on TimelineEventCard" but don't document that WeekTimelineView bypasses that card entirely.
**How to avoid:** Fix is to the inline color literals in `weekEventView()` directly, not via the TimelineEventCard parameter.
**Warning signs:** Attempting to add `isTimeBlock:` to a call site that doesn't exist in WeekTimelineView.

### Pitfall 3: CALV-02 Checkbox May Already Be Correct

**What goes wrong:** Plan adds a checkbox tick to a line that already has `[x]`.
**Why it happens:** Phase 2 VERIFICATION.md noted "CALV-02 checkbox not ticked" but REQUIREMENTS.md shows `[x]` already (possibly fixed between Phase 2 and 3).
**How to avoid:** Verify current checkbox state before writing a task to change it. If already `[x]`, task should confirm it and close without modification.
**Current state:** REQUIREMENTS.md line 12 currently reads `- [x] **CALV-02**: User can view events in a week calendar view` — already correct.

### Pitfall 4: 03-02-SUMMARY.md Frontmatter Not at Top

**What goes wrong:** YAML frontmatter block added in wrong location — not at the very top of the file between `---` delimiters.
**Why it happens:** File already has YAML frontmatter; adding `requirements-completed` in the wrong place.
**How to avoid:** Add the field inside the existing `---` frontmatter block that already exists at the top of 03-02-SUMMARY.md (between lines 1 and 48). Do not create a second YAML block.
